## Input

Read the user's message for scope and focus. No required fields — running with no arguments analyses all available profiling data.

## Goal

Read profiling results from `logs/profiling/`, analyse performance hotspots in backend endpoints and Celery tasks, and propose targeted optimization areas. After each run, update `.github/skills/profiling/findings.md` with durable findings.

## Sources of Profiling Data

1. **Backend API profiles** — generated by `make profile CMD="..."` or pytest-benchmark runs. Written to `logs/profiling/backend/<timestamp>/`.
2. **Celery task profiles** — pyinstrument traces from task runs; written to `logs/profiling/tasks/<timestamp>/`.
3. **Frontend Lighthouse** — Lighthouse CLI reports written to `logs/profiling/frontend/<timestamp>/report.json`.
4. **Test suite timing** — `pytest --durations=20` output; written to `logs/profiling/test-timing/<timestamp>/`.

## Analysis Steps

### 1. Backend Endpoint Performance

- Identify the top slowest routes from benchmark results
- Flag routes with P95 > 200ms (candidates for caching or query optimization)
- Distinguish I/O-bound (DB queries, httpx calls) vs CPU-bound hotspots
- **Expected hot spots** (not bugs unless disproportionate):
  - `POST /products/{id}/scrape` — full HTTP fetch + parse; expected ~1-5s
  - `GET /products/{id}/prices` — DB query with pagination; flag if > 50ms

### 2. Celery Task Performance

- Analyse `scrape_product` task duration distribution
- Flag tasks taking > 10s consistently (check retry logic, scraper timeout settings)
- Identify redundant DB calls inside tasks (N+1 query patterns)
- Check for tasks that should be subtasks but run serially

### 3. Frontend Performance (Lighthouse)

- Flag First Contentful Paint > 2s
- Flag Time to Interactive > 3.5s
- Identify large bundle chunks (> 250KB) for code-splitting
- Note any render-blocking resources

### 4. Test Suite Performance

- Identify the top 10 slowest tests
- Flag tests taking > 2s (candidates for mocking or DB fixture optimization)
- Look for integration tests that could be unit tests with proper mocking

## Operating Rules

1. **Always read the latest profiling data first** — do not analyze stale results
2. **Be specific** — reference exact function names, file paths, and line numbers
3. **Quantify impact** — estimate the time savings for each proposed optimization
4. **Rank by ROI** — order proposals by expected improvement relative to effort
5. **Verify before proposing** — read the actual source code of flagged functions before suggesting changes
6. **Create actionable todos** — each optimization proposal should have a concrete next step

## Quick Commands

```bash
# Run backend benchmarks
cd backend && uv run pytest tests/ --benchmark-only --benchmark-json=logs/profiling/backend/latest.json

# Profile specific endpoint (requires running stack)
make profile CMD="curl -s http://localhost:8000/api/v1/products"

# Frontend Lighthouse (requires running frontend)
npx lighthouse http://localhost:5173 --output json --output-path logs/profiling/frontend/latest.json

# Test suite timing
cd backend && uv run pytest --durations=20 -q 2>&1 | tee logs/profiling/test-timing/latest.txt
```

## Output

Produce a structured analysis with:

1. **Executive summary** — overall performance health, top 3 issues
2. **Detailed findings** — per-profile breakdown with specific function/line references
3. **Optimization proposals** — ranked by expected impact, each with:
   - Description of the issue
   - Affected file(s) and function(s)
   - Proposed fix approach
   - Estimated time savings
   - Implementation effort (low/medium/high)
4. **Action items** — concrete next steps as todos
